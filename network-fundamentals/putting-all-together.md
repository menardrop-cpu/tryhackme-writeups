## Objective

Understand how all the individual components of the web work together when a browser requests a website: DNS resolution, HTTP protocol, infrastructure components, and web server internals.

---

## Task 1 — Putting It All Together

When you request a website, two core mechanisms are at play:

* **DNS** : resolves the domain name into an IP address
* **HTTP** : the protocol used to communicate with the web server

The server then returns HTML, CSS, JavaScript and images that the browser uses to render the page.

---

## Task 2 — Other Components

Beyond DNS and HTTP, several infrastructure components sit between the user and the web server.

### Load Balancer

Distributes incoming traffic across multiple backend servers. Two main purposes:

* **Scalability** : handles high traffic volumes
* **High availability** : provides failover if a server goes down

Common load balancing algorithms:

| Algorithm | How it works |
|---|---|
| Round-robin | Sends requests to each server in turn |
| Weighted | Sends to the least busy server |

Load balancers also run **health checks** — periodic requests to each backend server to verify it is responding correctly. If a server fails a health check, it is removed from the pool until it recovers.

### CDN (Content Delivery Network)

Hosts static files (JavaScript, CSS, images, videos) across thousands of servers worldwide. When a user requests a static resource, the CDN routes the request to the geographically nearest server, reducing latency.

### Database

Web servers communicate with databases to store and retrieve dynamic data. Common databases: MySQL, MSSQL, MongoDB, PostgreSQL.

### WAF (Web Application Firewall)

Sits between the user's request and the web server. Its role:

* Analyses requests for known attack signatures (SQLi, XSS, etc.)
* Distinguishes real browsers from bots
* Enforces **rate limiting** — limits the number of requests per IP per second
* Drops malicious requests before they reach the web server

**Q: What can be used to host static files and speed up a client's visit to a website?**  
`CDN`

**Q: What does a load balancer perform to make sure a host is still alive?**  
`health check`

**Q: What can be used to help against the hacking of a website?**  
`WAF`

## Task 3 — How Web Servers Work

### What is a Web Server?

A web server is software that listens for incoming connections and responds using the HTTP protocol. Common web servers: Apache, Nginx, IIS, NodeJS.

A web server serves files from a **root directory** defined in its configuration:

| Software | Default root directory |
|---|---|
| Apache | `/var/www/html` |
| Nginx | `/var/www/html` |
| IIS | `C:\inetpub\wwwroot` |

### Virtual Hosts

A single web server can host multiple websites using **virtual hosts**. The server checks the `Host` header in the HTTP request and maps it to the correct root directory.

Example:
* `one.com` → `/var/www/one.com/`
* `two.com` → `/var/www/two.com/`

There is no limit to the number of virtual hosts on a single server.

### Static vs Dynamic Content

| Type | Description | Examples |
|---|---|---|
| Static | Files served exactly as stored on disk | HTML, CSS, images |
| Dynamic | Content generated in real time by backend code | User profiles, search results, dashboards |

### Backend Languages

Dynamic content is generated by backend code that runs on the server, not in the browser. Common backend languages: PHP, Python, Ruby, NodeJS, Perl.

Example of a simple PHP script:

```php
<?php
    echo "Hello " . $_GET["name"] . "!";
?>
```

A request to `http://example.com/hello.php?name=Pierre` would return:

```
Hello Pierre!
```

The browser only receives the rendered output — it never sees the backend source code.

## Full Request Flow — Step by Step

The complete sequence when a user types `tryhackme.com` in their browser:

| Step | Action | Component |
|---|---|---|
| 1 | User types the URL | Browser |
| 2 | Check local DNS cache for IP | Local cache |
| 3 | Query recursive DNS server | ISP / configured resolver |
| 4 | Query root server for authoritative DNS | Root DNS servers |
| 5 | Authoritative DNS returns the IP address | Authoritative DNS |
| 6 | Connect to web server on port 80 or 443 | TCP / TLS handshake |
| 7 | Request passes through the WAF | WAF |
| 8 | Request passes through the load balancer | Load Balancer |
| 9 | Web server receives the GET request | Web Server |
| 10 | Web application queries the database | Database |
| 11 | Browser renders the HTML into a webpage | Browser |

### Three phases

```
DNS Resolution (steps 2–5)
  → Find the IP address of the server

Security & Routing (steps 6–8)
  → Establish the connection, filter threats, distribute load

Processing & Rendering (steps 9–11)
  → Generate the response, retrieve data, display the page
```

## Vulnerability Surface Summary

Each component in the request chain introduces potential attack surface:

| Component | Attack vector | Mitigation |
|---|---|---|
| DNS | DNS spoofing, cache poisoning | DNSSEC |
| WAF | WAF bypass techniques | Regular rule updates, tuning |
| Load Balancer | Session fixation, IP leakage | Sticky sessions config, IP masking |
| Web Server | Directory traversal, misconfiguration | Hardening, patching |
| Database | SQL Injection | Prepared statements, least privilege |
| Frontend | XSS, HTML Injection | Output encoding, CSP |
